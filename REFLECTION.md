Мой проект — это агент, который помогает решить проблему хаоса в задачах. Основная идея заключалась в том, чтобы дать LLM возможность видеть задачи пользователя и напрямую менять их в таск-трекере TickTick. Я выбрал именно эту задачу, потому что сам каждый день пользуюсь подобными инструментами и хорошо понимаю, насколько быстро список задач может превратиться в неструктурированный набор записок, напоминаний и незавершённых задач. В проекте я реализовал 11 tools, которые позволяют агенту выполнять основные действия с задачами: создавать, изменять, перемещать, искать и завершать задачи в TickTick. Агент должен помогать на этапе Clarify в методике GTD. То есть его задача — находить слишком крупные или расплывчатые задачи, предлагать для них следующие действия, а также помогать разбивать их на более конкретные подзадачи. Таким образом, агент не просто хранит задачи, а помогает превращать хаотичные записи в понятный план действий.

В процессе разработки основной проблемой стало API TickTick. В нём Inbox отделён от обычных проектов и представлен как отдельная сущность. Из-за этого агент часто путался при перемещении задач и выборе проекта. На текущем этапе я частично решил эту проблему через создание alias id для входящих, чтобы агент мог более стабильно понимать, что пользователь имеет в виду под Inbox. Сейчас основные ошибки возникают в сценариях, где требуется вызвать несколько инструментов подряд. Например, найти задачу, изменить её описание, перенести в другой проект и добавить подзадачи. Быстрые модели, такие как qwen-turbo, хорошо подходят для плавного пользовательского опыта, но иногда хуже справляются с многошаговым планированием. Эту проблему можно решить двумя способами: добавить отдельного агента-оркестратора, который будет управлять последовательностью действий, или использовать более “думающую” модель, хотя это может снизить скорость работы.

Дальнейшее развитие проекта я вижу в переходе к системе из нескольких агентов. Один агент мог бы быть оркестратором, а два других отвечали бы за этапы Clarify и Organize по GTD. Также стоит добавить несколько разных workflow для Clarify, потому что сейчас используется только базовый сценарий просмотра задач и советов по реорганизации. Кроме того, важно реализовать tools для работы с тегами, так как GTD активно использует контексты, категории и разделение задач по условиям выполнения.
